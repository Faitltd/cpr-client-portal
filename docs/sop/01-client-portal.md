# Client Portal — SOP

**Audience: homeowners (CPR renovation clients). Route: `/dashboard`.**

---

## 1. Purpose and users

The Client Portal is where a homeowner follows their renovation. On one scrolling page they see today's site status, project progress, tasks, financials, invoices to pay, change-order requests, contracts to sign, photos, documents, project emails, and an AI assistant scoped to their project.

A homeowner gets access when their Zoho CRM Contact is attached to a Deal in an active stage. An admin sync (see [Admin Portal → Clients](04-admin-portal.md)) creates a Supabase `clients` row for them with `portal_active = true`.

## 2. How they log in

- **Entry:** `/auth/client` (client-only) or `/auth/portal` (shared front door that tries admin → designer → client → trade).
- **Username:** their email. **Password:** their phone number on file is the seed password.
- **Self-healing:** if the stored password doesn't verify, the system falls back to matching the typed value against the phone number, and can re-pull the contact live from Zoho to re-seed the login. This is why a client can almost always get in with their phone number.
- **No magic link, no "forgot password."** If a client is locked out, an admin resets them from `/admin/clients`.
- **Session:** a `portal_session` cookie (httpOnly, strict, 7-day). Sessions do not re-check `portal_active`, so deactivation means removing the row/password.

## 3. Layout

A global header shows the Home CPR logo and a Log out link. The page title is "My Projects." There is **no tab bar** — it's one vertical scroll of collapsible sections. Data loads client-side after the page paints, so sections fill in progressively.

Only the **first project** drives the status, progress, assistant, checklist, tasks, and access sections. Financials, invoices, change orders, and emails aggregate across all the client's deals.

There are three render modes:

1. **No projects and no billing** → "No projects found."
2. **Billing exists but no active project card** (e.g., an On-Hold account) → a billing-only view showing Financial Summary and Invoices.
3. **Normal** → the full section list below.

## 4. Sections, in order

**Schedule a Project Review** — always-visible banner. Embeds Zoho Bookings ("15-minute review, Tue–Thu 4–5 PM"). Action: Pick a time.

**Project status — "Today on site"** — an AI-written positive daily summary plus up to **4** photo thumbnails with a lightbox. Built from Cliq field-update messages, completed Zoho Projects tasks, WorkDrive photos, and client-gallery photos submitted by the crew. Negative/problem language is stripped; the summary is generated by `gpt-4o-mini`. Starts expanded.

**Project progress** — animated donut of % tasks complete, plus done/remaining/total and per-stage bars for **Design, Demo, Build, Finish**. Derived from the same Zoho Projects tasks as the Tasks section.

**Ask CPR Assistant** — the only section open on load. Chat scoped to the client's deal, with quick prompts (status, recent changes, balance, scope, materials, documents). It answers from a vector store of the deal's data but **hides internal financials** and restricts to client-safe sources. Actions: type/send, quick prompts, New conversation.

**Home Building Checklist** — a static homeowner checklist grouped into phases with progress bars. Ticks autosave to Supabase (`client_checklist_progress`). Collapsed by default.

**Progress Photos** — curated crew/progress photos from WorkDrive and CRM attachments. "View Photos" opens the WorkDrive share or `/project/[id]/photos`. Distinct from photos a client attaches to a change order.

**Project Tasks** — a done/total pill and task rows (Complete/Open) from Zoho Projects. Read-only.

**Financial Summary** — three cards: Price, Invoiced, Balance. See §5.

**Invoices** — one card per invoice (number, status, date, total, balance) from Zoho Books. Buttons **Pay Now** and **View Invoice** appear when the Zoho payment/view URLs exist.

**Change Orders** — shows existing change-order line items and CO invoices, and lets the client request a new one. See §6.

**Email Updates** — a timeline of project emails (received/sent, sender, time, subject, summary); each row expands to the sanitized body. Below it, per-project email frequency (Daily / Weekly / None). Emails come from Zoho CRM; bodies fall back to a synced copy when the CRM detail is unavailable.

**Documents** — WorkDrive files grouped by folder, the Client Portal folder link, contracts (Zoho Sign), and CRM attachments. Only shown when a project exists.

**Update Access Info** — WiFi and Door Code fields, prefilled from the Deal and written back to Zoho CRM (`WiFi`, `Garage_Code`). Action: Save Access Info.

**Account** — set a new password (min 8, confirm) and Log out. No current-password re-entry required.

## 5. How the financial numbers work

Two independent data chains: CRM Deals produce the "projects" list; Zoho Books (keyed by a resolved customer id) produces the financials.

- **Price** = sum of Books estimates whose status is "counted" (accepted or invoiced). Sent/draft/declined/expired quotes count as $0. Shown only when greater than zero.
- **Invoiced** = sum of every invoice's total.
- **Balance** = Price − Invoiced when a quote exists; otherwise the summed open-invoice balance.

**Matching the client to a Books customer** happens in three ordered steps:

1. Use the stored `books_customer_id` on the client row if present.
2. Otherwise look the customer up by email.
3. Otherwise **safe auto-match**: search Books by the client's name and accept a match only if it is corroborated by phone or by billing zip + street number, and only if exactly one candidate passes. That result is saved back so it self-heals once.

If nothing resolves, the portal shows no financials. This is the mechanism behind the Lisbeth Ojemann fix: her login email did not match her Books email, so `books_customer_id` was set (and the auto-match now handles similar cases automatically).

## 6. The change-order request flow

The client picks a project (if they have more than one), describes the change, and optionally attaches up to **5** photos/videos. On submit, the server does several things (each after the first is best-effort):

1. Writes a Zoho CRM **Field_Updates** record (`updateType: change_order`), which triggers a Cliq workflow.
2. Creates a Zoho Books estimate (quote) draft.
3. Attaches the photos to that estimate.
4. Inserts a Supabase `change_orders` row (status `identified`).
5. Posts a Cliq notification to the deal channel.
6. Creates a Zoho CRM **Task** — "Review Change Order — {project}", High priority, due +2 days, linked to the Deal, **assigned to Mary Sue** and **mentioning Jeff** in the description.

The client sees "Change order request submitted. The office team has been notified," with warnings if the Books draft or Cliq post failed. The durable records are the Supabase row, the CRM Field_Updates record, and the review Task.

Note: there is **no estimated-cost field** on this form — it was removed. Mary Sue's owner id and Jeff's mention text are hardcoded in `src/lib/server/zoho-field-updates.ts`; changing staff requires a code edit.

## 7. Project detail and contracts

**`/project/[id]`** — a per-project page with header, overview (closing date, PM = Deal owner, scope), editable access info, contracts, documents, Zoho Projects tasks, change orders, and a timeline from CRM notes. Same change-order flow as the dashboard.

**`/project/[id]/photos`** — WorkDrive photos only (no upload here). If the Deal's link is an external WorkDrive share, the browser is redirected straight to WorkDrive; otherwise photos are listed grouped by date with a lightbox.

**Contracts (Zoho Sign)** — the Documents section lists sign requests where the logged-in email is a verified recipient (the app re-checks recipients per request because Zoho's own filter would leak other contracts). Actions: **Sign** (embedded Zoho signing session), **View**, and **View PDF** once complete. The PDF endpoint handles Zoho occasionally returning a ZIP by extracting the first PDF.

**`/decisions`** — a standalone page (not linked from the dashboard) showing client-assigned approvals with Approve/Decline buttons.

## 8. Data sources

Zoho CRM (Deals, Contacts, Notes, Emails, Field_Updates, Tasks), Zoho Books (estimates, invoices, contacts), Zoho Projects (tasks), Zoho WorkDrive (documents, photos), Zoho Cliq (daily update, change-order notifications), Zoho Sign (contracts), Zoho Bookings (review scheduler). Supabase: `clients`, `client_sessions`, `client_checklist_progress`, `change_orders`, `field_updates` + `trade-photos` storage, `approvals`, `bot_documents`. OpenAI for the daily summary and the assistant.

## 9. Gotchas

1. **Phone number always works as a password**, even after a custom one is set. Password changes don't disable it.
2. **No forgot-password flow** — recovery is the phone fallback or an admin reset.
3. **Dashboard shows only the first project** for status, progress, assistant, checklist, tasks, and access. Multi-deal clients see one project for those.
4. **The "billing-only" view triggers when the CRM deal lookup returns nothing** while Books still resolves — not strictly the On-Hold stage. A genuinely On-Hold deal still linked to the contact renders as a normal project.
5. **Two photo caps:** the daily-update gallery caps at 4; change-order uploads cap at 5.
6. **Price counts only accepted/invoiced quotes**, so it can read $0 while a "sent" quote is outstanding; Balance then falls back to open-invoice balance.
7. **Financials silently blank** if the Books customer can't be matched and there's no zip/phone to auto-verify. Set `books_customer_id` manually in that case.
8. **Several endpoints exist but aren't wired into the dashboard** (`snapshot`, `activity`, `pending`); Decisions lives only at `/decisions`.
